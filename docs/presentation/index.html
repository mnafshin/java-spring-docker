Dockerfile Decisions with Evidence

Docker engineering briefing for Spring Boot container delivery

Benchmark-driven choices + security non-negotiables

Agenda

  1. Decision model and benchmark method
  2. Scenario-by-scenario benchmark findings (7 metric scenarios)
  3. Security, JVM flags, and production policy

The core question

Not "Which Dockerfile looks more sophisticated?"

Which option improves cost/performance/reliability for this service, on this workload, with acceptable complexity?

Decision framework

  • Policy-driven (must enforce): non-root runtime and digest pinning
  • Metric-driven: BuildKit cache, stage layout, jlink/jdeps, AOT cache, JVM flags, base image, native image
  • Policy choices are defaults; metric choices need benchmark evidence

Benchmark methodology

  • Benchmark scenarios: 01, 02, 03, 04, 05, 06, 07 (metric-driven choices)
  • Metrics: build avg, startup avg/p95, image MB, success rate
  • Full profile: 10 runs standard scenarios, 15 runs AOT scenario
  • Native-vs-JVM long run evaluated separately (different objective)

Execution workflow with springdocker

springdocker doctor --project-root .
springdocker benchmark generate --project-root . --build-tool maven --java-version 25
springdocker benchmark run --project-root . --profile full --runner-arg=--skip-native

  • This is often the highest-ROI Docker optimization

03 · JLink + JDeps decision

  • jdeps computes required modules, jlink creates minimized runtime
  • For reflection/proxy/dynamic loading, merge a curated must-have.txt module list
  • with-jlink: 100.40 MB, startup 1,423 ms
  • without-jlink: 144.38 MB, startup 1,446 ms

↓ Press down for generated flow

03 implementation flow

# from must-have.txt (reflection/proxy/manual additions)
# jdk.crypto.ec
# java.naming
jdeps --print-module-deps app.jar > modules.txt
MODULES="$(cat modules.txt),$(paste -sd, must-have.txt)"
jlink --add-modules "$MODULES" --compress=2 --output /opt/custom-java

Use must-have.txt for modules jdeps cannot infer (reflection, dynamic loading, proxies).

03 Dockerfile example

ARG MUSTHAVE_MODULES="jdk.crypto.ec,java.naming" # from dockerfile.must_have_modules_file
RUN jdeps --print-module-deps app.jar > modules.txt
RUN MODULES="$(cat modules.txt),${MUSTHAVE_MODULES}" && \
    jlink --add-modules "$MODULES" --compress=2 --output /opt/custom-java
ENV JAVA_HOME=/opt/java

↓ Press down for workload-fit rule

07 workload-fit rule

  • Use native for cold-start and memory-constrained workloads
  • Use JVM for sustained high-throughput services
  • Do not choose native globally without throughput profiling

Complete JVM Dockerfile reference (springdocker)

# syntax=docker/dockerfile:1.7
FROM debian:bookworm-slim@sha256:... AS build
WORKDIR /app
RUN apt-get update && apt-get install -y --no-install-recommends openjdk-25-jdk-headless passwd && rm -rf /var/lib/apt/lists/*

COPY mvnw pom.xml ./
COPY .mvn ./.mvn
RUN chmod +x mvnw
COPY src ./src
RUN --mount=type=cache,sharing=locked,target=/root/.m2 \
    ./mvnw -B -q package -DskipTests

FROM build AS jre-builder
WORKDIR /jre
COPY --from=build /app/target/*.jar app.jar
RUN jdeps --ignore-missing-deps --recursive --multi-release 25 --print-module-deps app.jar > modules.txt
ARG MUSTHAVE_MODULES="jdk.crypto.ec,java.naming" # from dockerfile.must_have_modules_file
RUN set -eux; \
    MODULES=$( (tr ',' '\n' < modules.txt; printf '%s\n' "$MUSTHAVE_MODULES" | tr ',' '\n') \
      | sed 's/^[[:space:]]*//;s/[[:space:]]*$//' | grep -v '^$' | sort -u | paste -sd, -); \
    jlink --add-modules "$MODULES" --strip-debug --no-man-pages --no-header-files --compress=2 --output /opt/java

FROM debian:bookworm-slim@sha256:... AS aot-trainer
RUN apt-get update && apt-get install -y --no-install-recommends passwd && rm -rf /var/lib/apt/lists/*
COPY --from=jre-builder /opt/java /opt/java
ENV JAVA_HOME=/opt/java
ENV PATH="${JAVA_HOME}/bin:${PATH}"
WORKDIR /app
COPY --from=build /app/target/*.jar app.jar
RUN java -XX:AOTCacheOutput=app.aot -Dspring.context.exit=onRefresh -jar app.jar && test -f app.aot

FROM debian:bookworm-slim@sha256:...
RUN apt-get update && apt-get install -y --no-install-recommends passwd && rm -rf /var/lib/apt/lists/*
RUN groupadd --system --gid 1001 javauser && useradd --system --uid 1001 --gid 1001 --no-create-home --shell /usr/sbin/nologin javauser
RUN install -d -o 1001 -g 1001 -m 755 /app && install -d -o 1001 -g 1001 -m 1777 /tmp
WORKDIR /app
COPY --from=build --chown=1001:1001 /app/target/*.jar app.jar
COPY --from=jre-builder /opt/java /opt/java
COPY --from=aot-trainer --chown=1001:1001 /app/app.aot app.aot
ENV JAVA_HOME=/opt/java
ENV PATH="${JAVA_HOME}/bin:${PATH}"
EXPOSE 8080
EXPOSE 8081
USER 1001
ENTRYPOINT ["java","-XX:AOTCache=app.aot","-XX:MaxRAMPercentage=75","-XX:+ExitOnOutOfMemoryError","-Djava.io.tmpdir=/tmp","-jar","app.jar"]

Security deep dive (what is still required)

  • jlink reduces runtime surface but does not patch vulnerable dependencies
  • Add SBOM + vulnerability scanning in CI
  • Use digest-pinned deploy refs and signed artifacts where possible
  • Network-policy protect management endpoints; do not expose by convenience
