Dockerfile Decisions with Evidence

Spring Boot container engineering for production systems

Afshin Amini

Who this talk is for

  • Backend engineers shipping Spring Boot on Kubernetes
  • Platform/DevOps teams maintaining CI image pipelines
  • Tech leads balancing speed, cost, and operational risk

The actual problem

Most teams can build a Java container.

Few teams can defend each Dockerfile decision with data.

Production impact of weak Docker decisions

  • longer CI cycles and slower release cadence
  • larger pull times during deploy/rollback
  • higher runtime memory and startup variance
  • larger attack surface and weaker supply-chain posture

Decision model used in this project

Decision typeHow we decide
Policy decisionsAlways enforce (non-root, pinned digests, supply-chain checks)
Performance decisionsBenchmark before adopting (cache, jlink, AOT, JVM flags, base image)

Methodology (what makes results credible)

  • scenario-based comparisons for each architectural choice
  • multiple runs per variant (full profile)
  • startup p95 and confidence-aware interpretation
  • regression checks integrated in CI

Threats to validity (explicitly acknowledged)

  • host/environment differences change absolute numbers
  • service workload profile changes winner strategies
  • cold-start and throughput goals can conflict

Use findings as decision guidance, not universal laws.

Benchmark execution workflow

springdocker doctor --project-root .
springdocker benchmark generate --project-root . --build-tool maven --java-version 25
springdocker benchmark run --project-root . --profile full --runner-arg=--skip-native
springdocker benchmark analyze --project-root . benchmarks/03-custom-jre-jlink/results/raw.csv --format table

Scenario 01 — Stage layout

VariantImage MBBuild avgStartup avg
specialized-multi-stage100.40645 ms1,428 ms
simple-two-stage131.544,135 ms1,240 ms

Decision: Scenario 01

Adopt specialized stages as default.

Startup can improve with simpler layout, but build and image penalties are too high for most pipelines.

Scenario 02 — BuildKit cache

with-cache build avg: 622 ms

without-cache build avg: 3,410 ms

Highest ROI optimization in CI.

Decision: Scenario 02

BuildKit cache should be ON by default.

Disable only on runners/platforms that cannot support deterministic cache behavior.

Scenario 03 — JLink/JDeps

  • with-jlink: 100.40 MB, startup 1,423 ms
  • without-jlink: 144.38 MB, startup 1,446 ms

Decision: Scenario 03

Use JLink/JDeps with curated must-have modules.

Skip only when compatibility/tooling blockers are proven.

Scenario 04 — AOT cache

  • with-aot startup avg/p95: 1,429 / 1,462 ms
  • without-aot startup avg/p95: 1,509 / 1,730 ms

Decision: Scenario 04

Adopt only when startup tail is a priority and improvement is repeatable.

Scenario 05 — JVM flags

Quick and full runs produced different winners.

No universal tuning profile.

Decision: Scenario 05

Ship safe baseline flags first.

Apply aggressive tuning only after service-specific full-profile benchmarking.

Scenario 07 — Native vs JVM

MetricNativeJVM
Startup308 ms2,025 ms
Image size55.7 MB130.5 MB
Build time~635 s~9 s
Throughput (CPU-heavy)~1.0 req/s~27.7 req/s

Decision: Scenario 07

Choose native for cold-start-sensitive workloads.

Choose JVM for sustained throughput-heavy services.

Policy baseline (non-negotiable in production)

  • digest-pinned base images
  • non-root runtime user
  • supply-chain scanning + SBOM
  • orchestrator-level readiness checks

What to do Monday morning

  1. Enforce policy baseline (security + provenance)
  2. Enable BuildKit cache in CI
  3. Run full-profile benchmark on one real service
  4. Adopt only changes that show repeatable p95/cost gain

Closing

Standardize policy. Benchmark performance. Choose deliberately.
