Biggest measurable win: BuildKit cache mount

RUN --mount=type=cache,sharing=locked,target=/root/.gradle \
    ./gradlew --no-daemon dependencies -q
  • with cache: build avg 917 ms
  • without cache: build avg 7187 ms
  • Same image size (103.02 MB), same success rate
  • CI impact is huge when cache is shared via buildx cache backend

Multi-stage specialization pays off

  • Specialized multi-stage: 103.02 MB, build avg 607 ms
  • Simple two-stage: 134.56 MB, build avg 7178 ms
  • Startup roughly comparable; image and build cost are not

Custom jlink runtime is a clear win

  • with jlink: 103.02 MB, startup avg 1364 ms
  • without jlink: 147.04 MB, startup avg 1435 ms
  • Recommendation: keep jdeps + jlink unless platform constraints block it

JEP 483 AOT cache: benchmark by your environment

  • This run: without-aot-cache startup avg 1222 ms vs 1288 ms
  • Difference is small and sensitive to host/cache state
  • AOT cache remains valuable for cold-start-heavy workloads
  • Rule: enable only when measured in your real deployment context

Policy decisions vs metric decisions

  • Policy-driven: digest pinning, non-root runtime, readiness healthcheck
  • Metric-driven: build cache, stage layout, jlink, base image, AOT
  • This distinction prevents unsafe trade-offs for tiny speed gains

What to adopt now

  1. Keep digest pinning + non-root + readiness healthcheck
  2. Keep specialized multi-stage + BuildKit cache mount
  3. Keep jlink runtime unless portability issues appear
  4. Choose base image by org context (size vs support model)
  5. Benchmark AOT/JVM flags in production-like memory limits before deciding

Takeaways by audience

  • Dev team: own Dockerfile performance and reproducibility (cache mount, multi-stage, jlink)
  • SRE team: enforce policy invariants (non-root, readiness health semantics, digest pinning)
  • Platform team: provide shared CI build cache and benchmark pipelines as a platform capability
  • Everyone: treat AOT/JVM/base-image choices as benchmark decisions, not ideology
