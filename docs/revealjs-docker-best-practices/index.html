Biggest measurable win: BuildKit cache mount

RUN --mount=type=cache,sharing=locked,target=/root/.gradle \
    ./gradlew --no-daemon dependencies -q
  • with cache: build avg 622 ms
  • without cache: build avg 3,410 ms (5.5× slower)
  • Same image size (100.40 MB), same success rate (100%)
  • CI impact is huge when cache is shared via buildx cache backend

Multi-stage specialization pays off

  • Specialized multi-stage: 100.40 MB, build avg 645 ms, startup avg 1,428 ms
  • Simple two-stage: 131.54 MB, build avg 4,135 ms, startup avg 1,240 ms
  • Startup is slightly faster on simple layout, but image size and build cost are dramatically worse
  • Verdict: specialized multi-stage wins on image size (primary metric) — 31 MB saved

Custom jlink runtime is a clear win

  • with jlink: 100.40 MB, build avg 607 ms, startup avg 1,423 ms
  • without jlink: 144.38 MB, build avg 1,140 ms, startup avg 1,446 ms
  • 44 MB image savings — direct reduction in registry storage and pull latency
  • Recommendation: keep jdeps + jlink unless platform constraints block it

Policy decisions vs metric decisions

  • Policy-driven: digest pinning, non-root runtime, readiness healthcheck
  • Metric-driven: build cache, stage layout, jlink, base image, JVM flags, AOT
  • This distinction prevents unsafe trade-offs for tiny speed gains

Security review: what actually reduces attack surface?

ChoiceSecurity valueWhy
Digest pinningHighPrevents silent tag retargeting; improves provenance and rollback confidence
jdeps + jlinkMediumFewer JDK modules, fewer binaries/docs, smaller runtime surface
Non-root + read-only FS + dropped caps + seccompHighDirectly constrains post-compromise blast radius
Split management portConditionalHelpful only when combined with network policy / auth
AOT cache / JVM tuning flagsLowMainly performance and operability, not hardening

jlink helps security, but only in a limited way

  • Security upside: fewer runtime modules means fewer classes, tools, docs and native components shipped
  • Current must-have list is already narrow: java.base, java.logging, java.management, jdk.crypto.ec, etc.
  • Important nuance: jlink reduces surface; it does not patch vulnerable app libraries
  • Review kept modules regularly — especially jdk.unsupported, which is often necessary but expands surface

Security caveats found in the current stack

  • Actuator is exposed on management port 8081; split-port design helps, but it is not access control by itself
  • Current app has no Spring Security layer; keep 8081 internal and protect it with cluster policy
  • Dependency review flagged Spring Boot 4.0.1 Actuator advisories; presentation baseline should upgrade to 4.0.4+ before production rollout
  • For the main Kubernetes-targeted image, prefer Kubernetes probes and omit Docker HEALTHCHECK entirely rather than spawning a second JVM or adding probe tools

What to adopt now

  1. Keep digest pinning, but also pin the deployment image reference by digest — not :latest
  2. Keep non-root + read-only root FS + dropped capabilities + seccomp
  3. Keep specialized multi-stage + BuildKit cache mount + jlink — these are proven wins
  4. Use debian-bookworm-slim as the current balanced default base image (87.64 MB)
  5. Use Kubernetes readiness/liveness probes; don't add Docker HEALTHCHECK to the main image
  6. AOT cache shows 80 ms startup improvement at 15 runs — worth enabling for restart-heavy services
  7. JVM flag tuning showed no benefit on this host (defaults-like won at 1,370 ms); re-measure under memory pressure
  8. Upgrade Spring Boot / Actuator patch level before calling the stack production-ready

Separate long-run benchmark: Native vs JVM

  • Mixed workload: /bench/read, /bench/cpu, /hello · 60m duration · 50 VUs
  • Native: startup 308 ms · image 55.7 MB · build 635 s
  • JVM: startup 2,025 ms · image 130.5 MB · build 9 s
  • Native: 6.6× faster cold-start, 57% smaller image, ~3× lower idle CPU (0.11 vs 0.34)
  • JVM: 27.7 req/s throughput vs native 1.0 req/s on this CPU-bound workload mix
  • Use native for cold-start / memory-constrained workloads; JVM for warm high-throughput services

Takeaways by audience

  • Dev team: own Dockerfile performance and patch hygiene; smaller images do not replace CVE remediation
  • SRE team: enforce non-root, seccomp, read-only FS, and digest-pinned deploy refs
  • Platform team: provide shared CI build cache, internal-only management access, and regular dependency/image scanning
  • Everyone: treat AOT/JVM/base-image choices as benchmark decisions, and treat security controls as separate non-negotiables
